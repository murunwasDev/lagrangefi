# lagrangefi — Best Practices & Patterns

This document defines the coding standards, architectural patterns, and rules for each module.  
Each rule is tagged: **MUST** (required, correctness/security risk if skipped) · **SHOULD** (strongly recommended) · **CONSIDER** (worth doing when time permits).

## 1. Cross-Cutting Rules

### 1.1 Service Boundaries — MUST

- `chain/` is a **dumb execution layer**. It receives requests, runs on-chain operations, and returns results. No strategy logic, no DB access, no scheduling.
- `api/` **owns all business logic**: when to rebalance, which wallet to use, recording events, calculating stats.
- `web/` talks only to `api/`. Never call `chain/` from the browser — the NetworkPolicy enforces this at the infra level but it must also be true architecturally.
- Never break this layering. If you are tempted to add business logic to `chain/`, add it to `api/` instead and make `chain/` accept the computed values as parameters.

> **TODO (`Routing.kt`):** The routing layer currently contains direct `transaction { }` blocks, raw `StrategyEvents.insert` / `ChainTransactions.insert` / `StrategyStats.update` calls, a `calcTicks()` business function, and complex financial calculations. All of this must move into service methods. Routing handlers must only extract inputs, call a service, and return the result.

### 1.2 Secrets — MUST

- **Never put secrets in ConfigMaps.** ConfigMaps are world-readable in the cluster. Passwords, JWT secrets, encryption keys, and bot tokens always go in k8s Secrets.
- **Never log secret values.** Mask wallet phrases, private keys, and JWT secrets in all log output and Telegram alerts.
- **Never commit secrets.** `.env` files are gitignored; k8s secrets are applied manually.
- **Never transmit wallet phrases or private keys without transport encryption.** The `walletPrivateKey` field travels from `api/` to `chain/` in the JSON body of every `/execute/rebalance`, `/execute/close`, `/mint`, and `/wallet/balances` request. Inside k8s, pod-to-pod traffic is unencrypted unless a service mesh with mTLS (Istio, Linkerd) is configured. The NetworkPolicy in §6.4 restricts *which* pods can connect but not the confidentiality of what they send.

> **TODO:** Either (a) add a service mesh for mTLS between `api/` and `chain/`, or (b) require `chain/` to authenticate callers with a shared `X-Api-Secret` header so an intercepted request without the secret cannot be replayed. Document whichever approach is chosen as the accepted compensating control.

### 1.3 Idempotency — MUST

- Every state-changing request from `api/` to `chain/` must include an `idempotencyKey` (UUID v4 generated by `api/`).
- `chain/` must reject duplicate keys with HTTP 409 before executing any on-chain operation.
- `api/` must record the idempotency key in `strategy_events.idempotency_key` (unique index) **before** sending the request to `chain/`. If `chain/` returns a duplicate error, `api/` must treat that event as already succeeded and look up the existing record.
- The current in-memory Set in `chain/` is a known gap (post-MVP: move to Postgres-backed store).

> **TODO (`Routing.kt:336,370`):** The `DELETE /strategies/{id}` handler uses `"close-$strategyId-${System.currentTimeMillis()}"` as the idempotency key — a millisecond timestamp, not a UUID v4. This means two rapid calls get unique keys and the close executes twice on-chain. Additionally, the DB event is written *after* the chain call, not before. Fix: generate a UUID v4, insert the `strategy_events` row with `status = "pending"` first, then call `chain/`.

### 1.4 Error Handling Hierarchy — MUST

| Layer       | Recoverable errors                        | Fatal errors                         |
|-------------|-------------------------------------------|--------------------------------------|
| `chain/`    | Return `{ success: false, error: "..." }` | Let Fastify catch; return HTTP 500   |
| `api/`      | Catch + record `failed` event in DB; alert Telegram | Catch + return HTTP 5xx |
| `web/`      | Show inline error state; never swallow    | Redirect to error page               |

Never throw from inside a `catch` block without intentionally escalating. Never swallow errors silently.

> **TODO (`Routing.kt:345`):** The `DELETE /strategies/{id}` handler wraps the entire `chainClient.close()` call in `catch (_: Exception) { /* non-fatal */ }`. A silently failed close leaves the LP position open on-chain with no scheduler watching it — funds are at risk and the user is unaware. This must be treated as a recoverable error: record a `failed` event to DB and send a Telegram alert. Only the post-close stats/snapshot enrichment (pool price fetch, `recordClose()`) is truly non-fatal.

### 1.5 Observability — SHOULD

- Use structured log lines (`key=value` pairs or JSON), not concatenated strings.
- Every rebalance cycle must log: `strategyId`, `tickLower`, `tickUpper`, `currentTick`, `outcome` (skipped/executed/failed).
- Every HTTP request to `chain/` must log: `method`, `path`, `statusCode`, `durationMs`.
- Use correlation IDs: generate a `requestId` per rebalance cycle; pass it to `chain/` and include in all logs for that cycle.

> **TODO (`ChainClient.kt`):** The `requestId` is generated in `api/` but never forwarded to `chain/`. Add an `X-Request-Id` header to every `ChainClient` request so `chain/` logs can be correlated to the triggering rebalance cycle.

**SHOULD:**
- Emit Prometheus metrics from `api/` and `chain/`: rebalance count (success/failure), chain service response latency (`durationMs`), active strategy count, time-in-range percentage. Add a Prometheus scrape annotation (`prometheus.io/scrape: "true"`) to both pod specs in `k8s/base/`. Without metrics, the only way to spot a degraded system is reading logs.

### 1.6 Decimal & Financial Math — MUST

- **Never use floating-point for token amounts or on-chain values.** Use `BigDecimal` (Kotlin) or `bigint` (TypeScript/viem) for raw ERC-20 amounts.
- The only place floats are acceptable is human-readable display formatting and swap ratio estimation (where precision loss is bounded and intentional — see `swap.ts`).
- Store fee amounts and gas costs as raw strings or `Long` (wei), not as floats or `Double`.
- `rangePercent` and `slippageTolerance` are stored as `Double` by design — they are configuration inputs to estimation math, not raw on-chain values. This is the intended exception to the rule above.

## 2. apps/api — Kotlin + Ktor

### 2.1 Configuration — MUST

**Pattern:** Hoplite typed config — already in use. Keep it.

```kotlin
// Good — typed, fails fast at startup
data class AppConfig(
    val database: DatabaseConfig,
    val jwt: JwtConfig,
    val chain: ChainConfig,
)
```

- **MUST:** All required env vars must be declared as non-nullable fields in the config data classes. Hoplite throws at startup if they're missing — that is the desired behavior.
- **MUST NOT:** Use `System.getenv("KEY") ?: ""` inline in business logic. All config access goes through `AppConfig`.
- **SHOULD:** Group related settings in nested data classes (`DatabaseConfig`, `JwtConfig`, etc.), not a flat list.

### 2.2 Database — MUST

**Pattern:** Exposed ORM with HikariCP. All DB access inside `transaction { }` blocks.

**MUST:**
- Every write path (create, update, delete) must be inside a `transaction { }` block.
- **Always check ownership before reading or writing a row.** The pattern is:
  ```kotlin
  // Correct — single query, atomic auth + fetch
  val strategy = transaction {
      Strategies.selectAll()
          .where { (Strategies.id eq strategyId) and (Strategies.userId eq userId) }
          .firstOrNull() ?: throw NotFoundException("Strategy not found")
  }
  ```
  Never do two separate queries (one to check ownership, one to fetch the row) — the current `getStats()` bug (`StrategyService.kt:333-348`) is a result of this.

> **TODO (`StrategyService.kt:333-348`):** `getStats()` runs an ownership check on line 335, then re-fetches the same row without `userId` on line 338. Fix by using a single query with both predicates and passing the result row forward.

> **TODO (`StrategyService.kt:366-424`):** `getEventHistory()` has the same double-query pattern. Additionally, for each event it runs separate queries for `RebalanceDetails` and `ChainTransactions` — up to 150 queries for the default 50-event limit. Replace with a JOIN or batch `WHERE strategyEventId IN (...)` fetches.

- All foreign-key columns that appear in `WHERE` clauses must have an explicit `.index()` on the column definition in `Tables.kt`:
  ```kotlin
  val userId = integer("user_id").references(Users.id).index()
  ```
- Use `uniqueIndex()` for columns that must be unique (e.g., `idempotency_key`, `wallet.user_id`).

> **TODO (`Tables.kt:26`):** `Strategies.userId` has no `.index()`. Every `listForUser`, `findById`, and `stop` call does a full table scan.

- **Never re-query data inside a write function that the caller already holds.** Pass it as a parameter instead. A function that writes to DB should not also read unrelated rows to complete its work.

> **TODO (`StrategyService.kt:291-292`):** `recordRebalanceEvent()` re-fetches `token0Decimals`/`token1Decimals` from `Strategies` inside a write transaction. The `StrategyRecord` is available at the call site in `UniswapStrategy.execute()`. Pass decimals as parameters.

- **`StrategyEvents.status` must have a canonical set of values defined as constants.** The allowed values are `pending`, `in_progress`, `success`, `failed` — but the table comment (`Tables.kt:83`) lists only three, and the fourth (`in_progress`) is scattered as a raw string across multiple files.

> **TODO (`Tables.kt:83`, `UniswapStrategy.kt`, `Routing.kt`):** Add a `StrategyEventStatus` object or sealed class with the four string constants, and replace all raw string literals. Update the table comment to list all four values.

- **Define a data retention policy for event and transaction tables.** `strategy_events`, `chain_transactions`, `rebalance_details`, and `strategy_snapshots` grow unbounded. A strategy polling every 60 seconds produces ~525 k event rows per year. Add a `CONSIDER` note with a target retention window (e.g., 90 days for events, 1 year for snapshots) and a cleanup cron job strategy.

**SHOULD:**
- Use Exposed's `batchInsert` for bulk inserts instead of looping `insert {}`.
- Avoid nested `transaction { }` blocks — Exposed reuses the outer transaction but it signals confused ownership. Prefer composing functions that assume a transaction is already in scope.
- Use `SchemaUtils.createMissingTablesAndColumns()` on startup only — never `SchemaUtils.drop()` in production code.

**CONSIDER:**
- Add a transaction timeout: wrap long-running transactions in `transaction(transactionIsolation = ..., repetitionAttempts = 3) { }` with retry logic for deadlocks.

### 2.3 Authentication & Authorization — MUST

**Pattern:** JWT HS256, 24h expiry, `JwtConfig` singleton.

**MUST:**
- Every protected route must call `call.getUserId()`. There must be no route under `/api/v1/*` or `/me/*` that skips this.
- Replace the `!!` non-null assertion in `getUserId()` (`JwtConfig.kt:55`) with explicit error handling:
  ```kotlin
  fun ApplicationCall.getUserId(): Int =
      principal<JWTPrincipal>()
          ?.payload?.getClaim("userId")?.asInt()
          ?: throw UnauthorizedException("Missing or invalid token")
  ```

> **TODO (`JwtConfig.kt:56`):** Current implementation uses `principal<JWTPrincipal>()!!` — throws `NullPointerException` instead of a clean 401.

- `UnauthorizedException` must map to HTTP 401 in `StatusPages.kt`.
- Authorization (does this user own this resource?) must happen **at the service layer**, not in routing. Routing extracts `userId`; services enforce ownership.

**SHOULD:**
- Differentiate `AuthenticationException` (401 — who are you?) from `AuthorizationException` (403 — you're not allowed) in `StatusPages.kt`.
- Add `iat` (issued-at) claim to JWTs to support token revocation if needed in the future.

> **TODO (`JwtConfig.kt:23-30`):** `iat` claim is not present. Add `.withIssuedAt(Date())` to `generateToken()`.

**MUST:**
- When registering an existing position (`POST /strategies`), verify that `PositionResponse.owner` matches the wallet address derived from the user's stored phrase. Any user can currently register any tokenId as their strategy — including positions belonging to other users' wallets.

> **TODO (`Routing.kt:279-309`):** No ownership check between the position's on-chain owner and the authenticated user's wallet address.

**CONSIDER:**
- Rate-limit `/auth/login` and `/auth/register` — even a simple in-memory counter (e.g., 10 req/min per IP) prevents brute-force.

### 2.4 Service Layer — MUST

**Pattern:** Services are the only place that touches the DB or calls `ChainClient`. Routing handlers are thin.

**MUST:**
- Service functions must be **pure with respect to side effects**: a function either reads or writes, not both unexpectedly. If `getStrategyWithStats()` triggers a chain call, that must be explicit in the function name and documented.
- Wrap all `ChainClient` calls in try/catch at the service layer. HTTP failures from `chain/` must not bubble up as 500s to the client — they are infrastructure errors, handle them specifically.
- Use sealed classes for service results where the caller needs to distinguish outcomes:
  ```kotlin
  sealed class RebalanceResult {
      data class Success(val event: StrategyEvent) : RebalanceResult()
      data class AlreadyInProgress(val eventId: Int) : RebalanceResult()
      data class OutOfRange(val tick: Int) : RebalanceResult()
      data class Failed(val reason: String) : RebalanceResult()
  }
  ```
  This forces callers to handle every case instead of checking magic strings.

**MUST:**
- Tick range calculation (`calcTicks` / `calculateNewRange`) must live in exactly one place. Currently `Routing.kt:63-71` and `UniswapStrategy.kt:204-212` implement the same Uniswap tick math with **different rounding behaviour** — `Routing` uses `floorDiv(upper, spacing) + 1`, `UniswapStrategy` uses plain integer division. This produces a starting tick range that differs from the rebalance tick range, making the position appear out-of-range immediately after creation.

> **TODO (`Routing.kt:63-71`, `UniswapStrategy.kt:204-212`):** Extract a single `calcTickRange(currentTick: Int, fee: Int, rangePercent: Double): Pair<Int, Int>` function into a shared location (e.g., a `StrategyMath.kt` utility file). Delete the duplicate.

**SHOULD:**
- Group related DB operations that must succeed together in a single `transaction { }`. For example, creating a strategy event + updating strategy_stats should be one transaction, not two.
- Validate inputs at the service layer, not just in routing. Routing validates HTTP shape; services validate business rules (e.g., "pollIntervalSeconds must be ≥ 10").

### 2.5 Strategy & Scheduling — MUST

**Pattern:** `StrategyScheduler` holds a `ConcurrentHashMap<Int, Timer>`. One `Timer` per active strategy.

**MUST:**
- `StrategyScheduler` must only instantiate strategies through the `ProtocolStrategy` interface — never hardcode `UniswapStrategy`. Use a registry or factory:
  ```kotlin
  // Bad — current state
  private val executor = UniswapStrategy(chainClient, telegram, strategyService)

  // Good
  private val executor: ProtocolStrategy = executorRegistry.get(strategy.type)
  ```

> **TODO (`StrategyScheduler.kt:30`):** `executor` is hardcoded to `UniswapStrategy`. This blocks adding any new protocol without modifying the scheduler.

- **Never run `runBlocking` inside a `fixedRateTimer` callback** for operations that take more than a few milliseconds. Timer threads are limited; blocking one during a 5-min rebalance starves other strategies. Use a dedicated coroutine scope instead:
  ```kotlin
  private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())

  fixedRateTimer(...) {
      scope.launch {
          try { executeOnce(strategy.id) }
          catch (e: Exception) { telegram.alert("Strategy ${strategy.id} error: ${e.message}") }
      }
  }
  ```

> **TODO (`StrategyScheduler.kt:48`):** `runBlocking { executeOnce(strategy.id) }` is called directly inside the timer callback. A 5-minute rebalance blocks the timer thread for that strategy for the entire duration.

- The in-progress guard is implemented via DB state (`StrategyEvents.status = "in_progress"`). This guard must not be duplicated — a single check at the top of `execute()` is sufficient. The check must use a query that is consistent with the idempotency key write (see §1.3).

> **TODO (`UniswapStrategy.kt:41-52` and `69-78`):** The same in-progress DB query runs twice in the same `execute()` call with no state change between them. Consolidate into a single check at the top of the function.

- **Define the allowed strategy status transitions explicitly.** The five statuses form a state machine; arbitrary transitions are bugs.

  | From | Allowed transitions | Trigger |
  |------|--------------------|----|
  | `INITIATING` | → `ACTIVE` | mint succeeds |
  | `ACTIVE` | → `PAUSED`, `STOPPED_MANUALLY`, `STOPPED_ON_ERROR` | user action / scheduler error |
  | `PAUSED` | → `ACTIVE`, `STOPPED_MANUALLY` | user action |
  | `STOPPED_MANUALLY` | — (terminal) | — |
  | `STOPPED_ON_ERROR` | — (terminal) | — |

  A `STOPPED_*` strategy cannot be restarted. The user must create a new strategy.

- **Define a startup recovery procedure for interrupted events.** On pod restart, `loadAndStartAll()` resumes only `ACTIVE` strategies. Events left in `pending` or `in_progress` from before the crash are permanently stuck and block future rebalances for that strategy (because `execute()` skips ticks when it finds an in-progress event).

> **TODO (`Application.kt` / `StrategyScheduler.kt`):** On startup, after loading active strategies, scan for `strategy_events` rows where `status IN ('pending', 'in_progress') AND triggered_at < NOW() - INTERVAL '10 minutes'` and mark them `failed` with `errorMessage = 'Interrupted by pod restart'`. This unblocks affected strategies automatically.

**SHOULD:**
- When a strategy timer fires and the strategy is no longer `ACTIVE` in the DB (e.g., stopped by another request while the timer was sleeping), skip the cycle and remove the timer — do not fail.

### 2.6 HTTP Client (ChainClient) — MUST

**Pattern:** Ktor `HttpClient` with two timeout variants — default and `longHttp` (5 min).

**MUST:**
- Always use `longHttp` for `/execute/rebalance` and `/execute/close` — these wait for on-chain confirmations.
- Always use `defaultHttp` for `/position`, `/pool-state`, `/wallet/balances` — fast read-only queries.
- Log the full request and response for every chain call at `DEBUG` level (mask wallet key).
- On HTTP 409 (idempotency conflict) from `chain/`, do **not** treat it as an error — look up the existing `strategy_events` record by idempotency key and return it as the result.

**SHOULD:**
- Add retry logic (max 2 retries, exponential backoff) for HTTP 503 and network timeouts from the chain service. Do **not** retry HTTP 500 — those indicate a logic error.

### 2.7 Wallet Encryption — MUST

**Pattern:** AES-256-GCM, 12-byte random IV per encryption, IV prepended to ciphertext.

**MUST:**
- The encryption key must be exactly 32 bytes. The current validation (`WalletService.kt:22`) is correct — keep it.
- Never log or return the decrypted wallet phrase. The phrase leaves the API process only as part of the HTTP request body to `chain/`.
- Validate that `WALLET_ENCRYPTION_KEY` decodes to exactly 32 bytes at startup (not lazily on first wallet save).

**SHOULD:**
- Add a key version field to the encrypted payload (`"v1:" + base64`) so future key rotation can identify which key version was used to encrypt each wallet. Without this, rotating the key requires re-encrypting all wallets at once.

### 2.8 Telegram Notifications — MUST

**Pattern:** Fire-and-forget async HTTP to Telegram Bot API.

**MUST:**
- **Never build JSON via string interpolation.** Use `kotlinx.serialization` or `buildJsonObject`:
  ```kotlin
  // Bad — current state (TelegramNotifier.kt:22)
  val body = """{"chat_id":"${settings.chatId}","text":"$message"}"""

  // Good
  val body = Json.encodeToString(TelegramMessage(chatId = settings.chatId, text = message))
  ```
  Error messages frequently contain quotes, newlines, and special characters that break raw JSON strings.

> **TODO (`TelegramNotifier.kt:22`):** JSON is built via string interpolation. Any error message containing `"`, `\n`, or `<` will produce malformed JSON or broken HTML in the Telegram message.

- All Telegram alerts must be non-fatal. If the Telegram call fails, log the error and continue. Never let a notification failure affect the rebalance outcome.

**SHOULD:**
- Truncate messages longer than 4096 characters (Telegram's limit) rather than letting the API call fail silently.
- Prefix messages with the strategy name: `[ETH/USDC 0.05%] Rebalanced at tick 201337`.
- **Rate-limit and deduplicate alerts per strategy.** If `chain/` is down and a strategy polls every 60 seconds, 10 minutes of downtime produces 10 identical alerts per strategy. Implement a cooldown: do not send the same alert class for the same strategy more than once per N minutes (e.g., 5 minutes for `chain service unavailable`, 1 minute for rebalance failures). Buffer or drop repeated alerts within the window.

### 2.9 Exception Handling — MUST

**Pattern:** Global `StatusPages` handler in `Plugins.kt`.

**MUST:**
- Distinguish exception types in `StatusPages`:
  ```kotlin
  exception<NotFoundException> { call, _ -> call.respond(HttpStatusCode.NotFound, ...) }
  exception<UnauthorizedException> { call, _ -> call.respond(HttpStatusCode.Unauthorized, ...) }
  exception<ValidationException> { call, _ -> call.respond(HttpStatusCode.BadRequest, ...) }
  exception<Throwable> { call, cause ->
      log.error("Unhandled exception", cause)
      call.respond(HttpStatusCode.InternalServerError, ...)
  }
  ```
  Catching all `Throwable` as a 500 hides `NullPointerException` and `ClassCastException` from programming errors.

> **TODO (`StatusPages.kt`):** Currently only `Throwable` is caught — every exception maps to 500, including `NotFoundException` (should be 404) and the `!!` NPE in `getUserId()` (should be 401).

- **Never expose stack traces to HTTP clients.** Log the full trace server-side; respond with a generic message.

**SHOULD:**
- Create a sealed class `AppError` hierarchy (`NotFound`, `Unauthorized`, `Forbidden`, `Conflict`, `ValidationError`) and map them in `StatusPages`. This prevents ad-hoc HTTP status decisions scattered across services.

## 3. apps/chain — TypeScript + viem

### 3.1 Configuration — MUST

**Pattern:** `config.ts` exports typed config object and `createWalletClientForKey()` factory.

**MUST:**
- Validate all required env vars at process startup (in `index.ts`), not lazily. Use a validation function:
  ```typescript
  function requireEnv(key: string): string {
      const val = process.env[key]
      if (!val) throw new Error(`Missing required env var: ${key}`)
      return val
  }
  ```
- `createWalletClientForKey()` must validate the key before passing to viem:
  - Private key: must match `/^0x[0-9a-fA-F]{64}$/`
  - Mnemonic: must pass `validateMnemonic()` from `viem/accounts`
  - Throw a descriptive error on invalid input — viem errors for invalid keys are cryptic.

**SHOULD:**
- Export `publicClient` and `createWalletClientForKey` only — never export a pre-initialized wallet client at module level. Wallet clients are per-request (containing the user's key) and must not be cached.

### 3.2 Route Handlers — MUST

**Pattern:** Fastify route handlers are thin — they validate input, call a service function, and return results.

**MUST:**
- **All route handlers must validate inputs before calling service functions.** Use Fastify's schema validation (`schema: { body: { ... } }`) or explicit guards:
  ```typescript
  // routes/pool.ts — current issue
  const { token0, token1, fee } = req.query
  // MUST validate before casting:
  if (!isAddress(token0) || !isAddress(token1)) {
      return reply.code(400).send({ error: 'Invalid token address' })
  }
  const feeNum = Number(fee)
  if (!Number.isInteger(feeNum) || ![100, 500, 3000, 10000].includes(feeNum)) {
      return reply.code(400).send({ error: 'Invalid fee tier' })
  }
  ```
- **Never cast query/body parameters to typed values without validation.** `token as \`0x${string}\`` without checking it is an address is a bug. Use viem's `isAddress()` guard.
- Idempotency check must be the **first operation** in all write handlers — before any DB read, before any chain call:
  ```typescript
  if (processedKeys.has(req.body.idempotencyKey)) {
      return reply.code(409).send({ error: 'Duplicate request' })
  }
  processedKeys.add(req.body.idempotencyKey)
  ```

**SHOULD:**
- Define Fastify JSON schemas for all request/response bodies in a `schemas/` directory and reference them in route definitions. This gives automatic validation and generates documentation.

### 3.3 On-Chain Operations — MUST

**Pattern:** `writeContract` → wait for receipt → parse logs. Tracked via `trackTx()` closure.

**MUST:**
- **Always wait for transaction receipt** (`publicClient.waitForTransactionReceipt`) before reporting success. A tx hash is not a confirmation.
- **Always parse return values from events, not from return values of `writeContract`.** Solidity return values are not reliably accessible via `writeContract`. Parse `IncreaseLiquidity`, `Transfer`, `Collect` event logs from the receipt.
- **Set `maxFeePerGas` and `maxPriorityFeePerGas` explicitly** or ensure the viem transport provides fee estimates. Never submit transactions without a gas limit.
- All contract calls must use a **per-request wallet client** (from `createWalletClientForKey()`), never a module-level singleton.

**SHOULD:**
- Wrap each `writeContract` call in a helper that retries once on `replacement transaction underpriced` errors with a bumped tip.
- Log the tx hash immediately after submission (before waiting for receipt) — if the process crashes while waiting, the hash is needed for manual recovery.

### 3.4 BigInt and Decimal Handling — MUST

**MUST:**
- All raw ERC-20 amounts and on-chain values must be `bigint`. Never assign them to `number`.
- **The only place floating-point math is acceptable is in `swap.ts` for ratio estimation.** The swap amount is then converted back to `bigint` via `BigInt(Math.round(...))`.
- When converting `bigint` to human-readable values for display or logging, use `formatUnits()` from viem, not custom division.

**MUST NOT:**
- `Number(someBigInt)` for any value that represents an on-chain amount (liquidity, tick, sqrtPrice, token amounts). Numbers lose precision above 2^53. Use `bigint` arithmetic throughout, or dedicated Uniswap math libraries.

**SHOULD:**
- Wrap the bigint ↔ float boundary in explicitly named conversion functions:
  ```typescript
  // Acceptable — the name makes the precision loss explicit
  function toHumanFloat(raw: bigint, decimals: number): number {
      return Number(formatUnits(raw, decimals))
  }
  ```

### 3.5 Error Handling — MUST

**Pattern:** Service functions return `{ success: boolean, ..., error?: string }`. They do not throw to the route layer.

**MUST:**
- Service functions (`rebalance.ts`, `mint.ts`, `close.ts`) must catch all errors and return `{ success: false, error: message }`. Never let an unhandled rejection reach the Fastify global handler from a known failure mode.
- The recovery attempt inside `rebalance.ts`'s catch block must itself be in a try/catch that does not throw:
  ```typescript
  try { /* ... rebalance ... */ }
  catch (err) {
      // Remove idempotency key to allow retry
      processedKeys.delete(req.body.idempotencyKey)
      let recoveryStatus = 'not-attempted'
      try {
          await recoverCollect(walletClient, position)
          recoveryStatus = 'recovered'
      } catch (recoverErr) {
          recoveryStatus = 'recovery-failed'
      }
      return reply.code(500).send({ success: false, error: String(err), recoveryStatus })
  }
  ```
- Remove the idempotency key **only** on errors that should be retried (e.g., RPC timeout, nonce collision). For business errors (e.g., insufficient balance), keep the key to prevent retries.

### 3.6 OpenAPI Contract — MUST

- The OpenAPI spec at `apps/chain/openapi.yaml` must exist and be kept up to date. It is the contract between `api/` and `chain/`. **Do not add or change endpoints without updating the spec first.**
- The spec must be validated in CI using `@redocly/cli` or `openapi-validator`.
- The spec must cover all request/response bodies, status codes, and error shapes for every endpoint.

### 3.7 Idempotency (Chain Service) — MUST / SHOULD

**MUST:**
- Every write endpoint (`/execute/rebalance`, `/execute/close`, `/mint`) must require an `idempotencyKey` in the request body and enforce it.
- HTTP 409 is the correct status code for a duplicate idempotency key.

**SHOULD (post-MVP):**
- Move the `processedKeys` Set to Postgres. Store `(idempotency_key, status, created_at)`. Query before processing. Add a TTL (e.g., 7 days) via a cleanup cron.
- This closes the gap where a pod restart allows double-execution.

## 4. apps/web — React + Vite + shadcn/ui

### 4.1 API Layer — MUST

**Pattern:** `api.ts` exports typed async functions that call `apiFetch`. Token management via localStorage.

**MUST:**
- Every `apiFetch()` call site must handle errors explicitly. Never rely on the global 401 redirect as the only error path:
  ```typescript
  // Bad — swallows non-auth errors
  fetchPosition().then(setPosition).catch(() => clearToken())

  // Good
  fetchPosition()
      .then(setPosition)
      .catch(err => {
          if (err.status === 401) { clearToken(); navigate('/login') }
          else setError(err.message ?? 'Failed to load position')
      })
  ```
- All API functions must have explicit TypeScript return types matching the shapes in `packages/shared`.
- Never import `api.ts` functions directly into components. Access them through a hook (e.g., `useApi()`) or pass them as props, so components remain testable.

**SHOULD:**
- Use `AbortController` for all `fetch` calls triggered from `useEffect` to cancel in-flight requests on component unmount:
  ```typescript
  useEffect(() => {
      const controller = new AbortController()
      fetchPosition({ signal: controller.signal }).then(setPosition)
      return () => controller.abort()
  }, [strategyId])
  ```

### 4.2 State Management — MUST

**Pattern:** React Context for auth state. Local `useState` for page-level data. No global state library.

**MUST:**
- **Never store derived values in state.** If a value can be computed from another state value, compute it in render or in a `useMemo`. Never `setDerivedValue(computeFromOtherState(x))` in a `useEffect`.
- Auth state lives in `AuthContext`. Every other piece of data (strategy, position, rebalances) is **fetched and held locally** by the page component that needs it.

**SHOULD:**
- Add a `useAsync` custom hook to standardize loading/error/data state for API calls:
  ```typescript
  function useAsync<T>(fn: () => Promise<T>, deps: DependencyList) {
      const [state, setState] = useState<{ data?: T; error?: string; loading: boolean }>({ loading: true })
      useEffect(() => {
          fn().then(data => setState({ data, loading: false }))
              .catch(err => setState({ error: err.message, loading: false }))
      }, deps)
      return state
  }
  ```
  This eliminates the pattern of three separate `useState` calls per async operation.

**CONSIDER:**
- Add TanStack Query (React Query) if the number of API calls grows significantly. It provides caching, background refresh, and deduplication without boilerplate.

### 4.3 Error States — MUST

**MUST:**
- Every page that fetches data must render **three states**: loading, error, and success. The loading and error states must be visible to the user — never render an empty page silently.
- Form submissions must show the server error message to the user (not just log it to console).

```tsx
// Pattern to follow
if (loading) return <Spinner />
if (error) return <ErrorBanner message={error} />
return <StrategyView strategy={strategy} />
```

### 4.4 Type Safety — MUST

**Pattern:** `types.ts` defines all domain types. `packages/shared` holds types used by both `api/` and `web/`.

**MUST:**
- The `StrategyStatus` type in `web/src/types.ts` must match the values the API actually returns. The current mismatch (`INITIATING | ACTIVE | STOPPED_MANUALLY | STOPPED_ON_ERROR` in web vs. `active | paused | stopped` in shared) is a bug. Pick one canonical set of values and use it everywhere — define it in `packages/shared` and import it in both `api/` and `web/`.
- Never use `as any` or type assertions (`as SomeType`) without a comment explaining why the type system can't express the constraint.
- All `apiFetch<T>()` call sites must supply `T` explicitly — never rely on inference from untyped JSON.

**SHOULD:**
- Add `zod` or `valibot` for runtime validation of API responses. TypeScript types are erased at runtime; if the API returns an unexpected shape, the app should fail with a clear error, not a silent `undefined`.

### 4.5 Forms & Input Validation — MUST

**MUST:**
- Validate user input **client-side before submitting** to the API. Minimum checks: required fields, numeric range (e.g., `pollIntervalSeconds >= 10`, `rangePercent > 0`), address format.
- Show inline validation errors per field, not a single banner after submission.
- Disable the submit button while a request is in-flight to prevent double-submission.

**SHOULD:**
- Use `react-hook-form` for all multi-field forms. It handles touched/dirty/error state correctly and avoids the manual `useState` proliferation seen in current pages.

### 4.6 Component Design — SHOULD

**SHOULD:**
- Keep components under ~150 lines. If a component is longer, split display logic from data-fetching:
  - `StrategyPageContainer` — fetches data, holds state, handles errors
  - `StrategyView` — pure display component, takes data as props, easy to test
- Utility functions (`tickToPrice`, `formatUsd`, `computeAPY`) must live in dedicated `utils/` or `lib/` files, not at the top of page components. They should be importable and testable independently.
- Replace hardcoded token address labels in `DashboardPage.tsx:8-12` with a lookup from the strategy's `token0Symbol`/`token1Symbol` fields returned by the API.

### 4.7 Performance — CONSIDER

**CONSIDER:**
- Memoize expensive calculations (`computeAPY`, `computeCompareToHold`) with `useMemo`.
- Use `React.lazy` + `Suspense` for page-level code splitting — the finance utilities are substantial.
- Add `loading="lazy"` to images and charts that are below the fold.

## 5. packages/shared — Shared Types

### 5.1 What Belongs Here — MUST

**MUST:**
- Any type that is sent over the wire between `api/` ↔ `web/` or `api/` ↔ `chain/` must be defined here. No duplicating types in `web/src/types.ts` that overlap with `packages/shared`.
- `StrategyStatus` must be defined once here and imported by both `api/` (as a string enum) and `web/` (as a TypeScript union).

**MUST NOT:**
- Put implementation code, business logic, or side-effectful code in `packages/shared`. Types and plain data-only interfaces only.
- Import from `apps/*` in `packages/shared`. The dependency direction is: `apps/*` → `packages/shared`, never the reverse.

### 5.2 Versioning — SHOULD

**SHOULD:**
- When changing a shared type that affects the wire format (adding a required field, renaming a field), bump the minor version of `packages/shared` and update all consumers together in the same commit.
- **CONSIDER** adding a version field to the main request/response types (e.g., `apiVersion: 'v1'`) so mismatches between deployed versions of `api/` and `web/` produce a clear error rather than silent data corruption.

### 5.3 Null Safety — MUST

**MUST:**
- All optional fields must be explicitly typed as `field?: T` or `field: T | null` — never leave off the `?` and rely on `undefined` behavior.
- Fields that are always present in responses must be non-optional (`field: T`, not `field?: T`). Optional fields signal to callers that they need null checks.

## 6. k8s — Kubernetes + Kustomize

### 6.1 Secrets — MUST

**MUST:**
- All secret values go in k8s `Secret` objects, never `ConfigMap`. The current separation (api-secret, postgres-secret) is correct — maintain it.
- Secrets must be created with `kubectl create secret generic` and applied manually to each namespace — never committed to git.
- Add a `secretRef` in each deployment's `envFrom` — do not inject individual secret keys as individual `env` entries unless the key name in the Secret must differ from the env var name.

### 6.2 Health Probes — MUST

**MUST:**
- Every Deployment and StatefulSet must define both `livenessProbe` and `readinessProbe`. Without them, Kubernetes marks pods as ready immediately after the container starts, before the app is actually accepting traffic.

```yaml
# Pattern for api and chain services
livenessProbe:
  httpGet:
    path: /health
    port: 3000
  initialDelaySeconds: 15
  periodSeconds: 20
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /health
    port: 3000
  initialDelaySeconds: 5
  periodSeconds: 10
  failureThreshold: 3
```

- The `/health` endpoint must return HTTP 200 only when the app is fully initialized (DB connected, scheduler running). It must return 503 during startup or on DB connection loss.

> **TODO (`Routing.kt:82-84`):** The current `/health` handler returns `{"status": "ok"}` unconditionally. It does not check DB connectivity or scheduler state. Kubernetes marks the pod ready immediately, including during DB connection loss.

### 6.3 Resource Requests & Limits — MUST

**MUST:**
- Every container must have both `requests` and `limits` defined. A container without limits can consume all node memory and trigger an OOM kill of other pods.
- Requests must reflect typical usage; limits should be ~2× requests to allow for spikes.

```yaml
# Recommended minimums for the api service (Kotlin JVM)
resources:
  requests:
    cpu: 250m
    memory: 512Mi    # JVM needs more than 256Mi
  limits:
    cpu: 1000m
    memory: 1Gi

# chain service (Node.js)
resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 256Mi
```

### 6.4 Network Policy — MUST

**MUST:**
- The existing ingress NetworkPolicy (only `api` can reach `chain` on port 3001) must never be removed. It is a security boundary, not just a recommendation.
- Add egress rules to restrict what the chain service can call outbound. At minimum, it should only be able to reach the configured Arbitrum RPC endpoint and Uniswap contracts — not arbitrary internet endpoints:
  ```yaml
  egress:
    - ports:
        - port: 443     # Arbitrum RPC (HTTPS)
  ```

### 6.5 Kustomize Structure — SHOULD

**SHOULD:**
- Base manifests (`k8s/base/`) must never contain environment-specific values (image tags, replica counts, resource overrides). Only overlays (`k8s/overlays/prod/`, `k8s/overlays/test/`) may override these.
- Use `namePrefix` or `nameSuffix` in overlays to prevent accidental cross-namespace resource conflicts.
- All overlay-level patches must use `patchesStrategicMerge` or `patches` with a named target — never copy the entire base manifest into the overlay.

### 6.6 Pod Security — SHOULD

**SHOULD:**
- Add a `securityContext` to all containers to limit privilege escalation:
  ```yaml
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    readOnlyRootFilesystem: true
    allowPrivilegeEscalation: false
  ```
- Mount secrets as read-only volumes where possible rather than environment variables — secrets in env vars are visible to all processes in the container.

### 6.7 Postgres StatefulSet — MUST

**MUST:**
- The Postgres StatefulSet must use a `PersistentVolumeClaim` with a storage class that supports volume snapshots.
- Enable Postgres's `synchronous_commit = on` (default) — do not override it to `off` for performance. Data loss risk for a financial app is unacceptable.
- Regularly test that PVC snapshots can be restored before production goes live.

### 6.8 Replica Safety — MUST

**MUST:**
- **`api/` must run with `replicas: 1` until distributed strategy locking is implemented.** The `StrategyScheduler` holds an in-process `ConcurrentHashMap<Int, Timer>`. With two replicas, both schedulers load the same active strategies, both timers fire at the same interval, and two independent rebalances execute with different idempotency keys — the idempotency protection covers request-level deduplication, not scheduler-level. On real money, this means double-execution.
- Enforce this with a `PodDisruptionBudget` (`maxUnavailable: 0`) alongside the Deployment so that rolling updates do not briefly run two replicas:
  ```yaml
  apiVersion: policy/v1
  kind: PodDisruptionBudget
  metadata:
    name: api-pdb
  spec:
    maxUnavailable: 0
    selector:
      matchLabels:
        app: api
  ```
- Before scaling `api/` beyond 1 replica, implement distributed strategy locking — for example, Postgres advisory locks acquired per `strategyId` at the start of each `executeOnce()` call. Only the replica that holds the lock executes; others skip.

## 7. CI/CD — GitHub Actions

### 7.1 Image Tags — MUST

**Pattern:** Images tagged `sha-<7char>`. Test overlay kustomization auto-updated on merge to main.

**MUST:**
- Never use `:latest` in any k8s manifest. Always pin to a specific SHA tag. This makes rollbacks deterministic.
- The `update-manifests` job in `ci.yml` must update the kustomization.yaml via `kustomize edit set image`, not `sed`. `kustomize edit` is idempotent and format-safe; `sed` is fragile.

### 7.2 Secrets in CI — MUST

**MUST:**
- All secrets used in CI (`GHCR_TOKEN`, cluster SSH key) must be stored as GitHub Actions Secrets (`${{ secrets.NAME }}`), never hardcoded or in environment files.
- The SSH private key for cluster access must be a **dedicated CI deploy key** with write access only to the deploy target — not a personal key or a key with admin access.
- Rotate CI secrets on a schedule or whenever team membership changes.

### 7.3 Build Caching — SHOULD

**SHOULD:**
- Use Docker BuildKit layer caching with the GHA cache backend (already in use):
  ```yaml
  cache-from: type=gha
  cache-to: type=gha,mode=max
  ```
- For the Kotlin API, cache the Gradle dependency cache between runs:
  ```yaml
  - uses: actions/cache@v3
    with:
      path: ~/.gradle/caches
      key: gradle-${{ hashFiles('apps/api/**/*.gradle.kts') }}
  ```

### 7.4 Deployment Safety — MUST

**MUST:**
- `deploy-test.yml` must run `kubectl rollout status` after `kubectl apply` and fail the workflow if the rollout does not complete within a timeout. Do not report success before pods are actually healthy.
- Never auto-deploy to `prod` namespace. Production deploys must always be manual (explicit `workflow_dispatch` or a manual approval gate).
- Add a workflow that validates all k8s manifests with `kubectl apply --dry-run=client -k` on every PR — this catches YAML errors before they reach the cluster.

### 7.5 OpenAPI Validation — SHOULD

**SHOULD:**
- Add a CI step that validates `apps/chain/openapi.yaml` against the actual Fastify routes using `@redocly/cli lint apps/chain/openapi.yaml`. This prevents silent API contract drift.

### 7.6 Dependency Updates — CONSIDER

**CONSIDER:**
- Enable Dependabot for npm and Gradle dependencies with auto-merge for patch updates. Financial applications need prompt security patches.
- Pin all GitHub Actions to a specific commit SHA (`uses: actions/checkout@a5ac7e5` not `@v4`) to prevent supply chain attacks via action tag hijacking.
